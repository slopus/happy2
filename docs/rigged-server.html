A small server with durable auth.

Rigged’s server is a configurable authentication authority and session validator. It intentionally keeps its public HTTP surface small, while making sessions, telemetry, and deployment boundaries explicit.

Fastify 5Drizzle + SQLite/libSQLRS256 JWTTOML configuredDocker ready
01 / Shape

Five pieces, one shared authority.

The process loads TOML, bootstraps managed secrets, applies Drizzle migrations, then assembles Fastify routes around an authentication service and a shared session store.

01TOML

Controls role, network boundary, database, JWT identity, and enabled mechanisms.

02Bootstrap

Loads adjacent .env. Missing keys and password pepper are generated once, persisted, then reused.

03Fastify

Serves a minimal root response and versioned /v0 auth + health routes.

04Auth service

Issues, verifies, refreshes, and revokes sessions. This is the reusable policy layer.

05Drizzle

Stores users, identities, sessions, links, OIDC state, and security events.

02 / Roles

One codebase, three deployment shapes.

server.role makes authentication issuance separable from API-side validation. A JWT alone is never sufficient authority: the session row must be active too.

all

Full server

Default for local development and simple deployments. Issues tokens, validates sessions, and exposes enabled auth routes.

auth

Dedicated auth

Hosts credentials, OIDC, and magic-link flows. Pair it with a separate API deployment sharing session storage and public key.

api

Validation-only API

Serves health + session validation, but omits credential-issuing endpoints. It needs the matching JWT public key.

03 / Auth

Three alternatives. One active.

Configuration allows exactly one method: password, magic link, or one OIDC provider. Clients call /v0/auth/methods to learn the selected method. Every successful path creates a database-backed session and signs a JWT whose sid identifies that session.

Password

Optional self-service

Registration is disabled unless signup_enabled = true. Passwords require 12+ characters, receive a unique scrypt salt, and use a persisted server pepper.

register → user → session → JWT

Magic link

SMTP only from env

A token is SHA-256 hashed before storage, expires after 15 minutes, and is atomically consumed. SMTP credentials live only in environment variables.

request → email → consume → session

OIDC

Standards-based login

Discovery, PKCE S256, state, nonce, token exchange, remote JWKS validation, and verified email are checked before identity linkage.

start → provider → callback → identity

Session rule. Tokens are RS256 signed and default to 30 days, but validation checks that the matching session row is present, unexpired, and not revoked. Refreshing signs a replacement token with the same sid and advances the database expiry.
04 / Routes

A deliberately narrow HTTP surface.

/ is status-only. Useful APIs are versioned at /v0. Clients should first call /v0/auth/methods to learn the one issuance method this deployment exposes.

05 / State

SQLite is the source of truth.

The default is a local SQLite file. A shared libSQL endpoint is supported for independently deployed auth and API instances. Drizzle applies bundled migrations at startup.

usersEmail and optional password hash.
oidc_identitiesProvider + subject → user mapping.
auth_sessionsSession owner, expiry, last seen, revocation.
auth_session_eventsIssued / refreshed / revoked telemetry.
auth_magic_linksOne-time token hashes and expiry.

file: / libsql:Switch database URL in TOML; name an env auth token when required.

Security event context

Issue, refresh, and revoke events retain proxy-aware client IP, forwarded chain, Cloudflare location headers when supplied, device (x-rigged-device), app version (x-rigged-app-version), and user agent. Forwarded values are trusted only through trusted_proxy_hops.

06 / Config

TOML controls behavior; secrets stay outside it.

The loader validates configuration at startup. OIDC client secrets and SMTP credentials are referenced by, and read from, environment variables rather than stored in TOML.

[server]
role = "all"                         # all | auth | api
host = "127.0.0.1"
port = 3000
public_url = "http://127.0.0.1:3000"
trusted_proxy_hops = 1

[database]
url = "file:rigged.db"               # or shared libSQL

[jwt]
issuer = "https://rigged.example.com"
audience = "rigged-desktop"
key_id = "2026-01"
expiry_days = 30

[auth.password]
enabled = true
signup_enabled = false

When an auth-capable server lacks mounted key files or externally supplied JWT variables, it generates a 3072-bit RSA keypair and writes base64 values to the .env beside the TOML. It similarly generates RIGGED_PASSWORD_PEPPER when password auth is enabled. Keep this file secret and durable.

07 / Run

Local startup is one command.

The root shortcut needs no TOML: it enables password registration/login, creates the local SQLite database in packages/server, applies migrations, and watches TypeScript sources.

pnpm dev:server

curl http://127.0.0.1:3000/
curl http://127.0.0.1:3000/v0/health

First run

Creates packages/server/rigged.db and packages/server/.env. Both are ignored by Git.

For Docker or custom deployment, copy rigged.example.toml, change network/database settings, and pass it with --config.
